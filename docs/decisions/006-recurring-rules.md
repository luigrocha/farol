# ADR-006: Recurring Rules Engine тАФ Replacing isFixed+copy

**Date**: 2026-05-07
**Status**: Implemented тЬЕ
**Area**: Domain ┬╖ Database ┬╖ UI

---

## Context

The only recurrence mechanism that existed was `fixedExpensePropagationProvider`: when opening a new month, expenses with `isFixed = true` were copied from the previous month. A SnackBar notified the user: "X fixed expenses copied from previous month."

Concrete problems:
- Variable amount (rent that increases) тЖТ the copy uses the old amount
- No alternative frequencies (weekly, semi-annual, annual)
- No pause/cancel without deleting
- No future start date
- Future expenses don't exist in DB тЖТ ForecastingEngine is blind to recurrings
- The UX of "copying expenses" is fragile: if the user doesn't open the month, the expenses don't exist

## Decision

**Replace the `isFixed + copy` mechanism with a real recurrence engine**:

1. `RecurringRule` тАФ defines the template of the recurring (frequency, amount, validity)
2. `RecurringOccurrence` тАФ concrete instances generated by the rule for a date range
3. `RecurrenceResolver` тАФ pure Dart algorithm that generates occurrences from a rule

Occurrences are generated proactively for the next 3 months when the app opens. The `ObligationEngine` of the `ForecastingEngine` reads `recurring_occurrences WHERE status = 'pending'` to calculate future obligations.

`fixedExpensePropagationProvider` is removed after migrating `isFixed = true` expenses to `RecurringRule`.

## Consequences

### Positive
- ForecastingEngine has visibility of future recurrings (without them, projection is off by R$2,000+/month)
- Support for real frequencies: monthly, weekly, biweekly, semi-annual, annual
- Pause with automatic reactivation date
- Variable amount: `amount_type = 'range'` uses the average, `'variable'` uses history
- Automatic detection of recurring patterns in history
- Real history: old occurrences marked as `paid` with link to the original expense

### Negative / Trade-offs
- Migration of `isFixed = true` expenses requires a script + manual review (some may not actually be recurring)
- `fixedExpensePropagationProvider` cannot be removed before RecurringEngine is stable in production
- The user needs to learn the new concept (RecurringRule vs manual Expense)

### Accepted Risks
- **Migration creates duplicate RecurringRules**: if the user had the same recurring in multiple months as separate expenses тЖТ grouping by `(category, store_description)` may create duplicates. Mitigated with a review step in post-migration onboarding.

## Alternatives Considered

### Improve `isFixed + copy` (add frequency, variable amount)
Add `recurrence_type` field to `Expense` and improve the copy mechanism.

**Discarded because**: The `Expense` model represents a past transaction тАФ not a recurrence template. Mixing both concepts in the same model increases technical debt. The copy mechanism never produces future occurrences, blocking forecasting.

### Use an RRULE library (RFC 5545)
Adopt an existing recurrence library (like the iCalendar RRULE standard).

**Evaluated positively but discarded for v1**: RRULE is very powerful for Farol's use cases (most are monthly/weekly/annual). A custom implementation covers 99% of cases with much less complexity. Can be adopted in v2 if RecurrenceResolver shows limitations.

## Success Criteria

- [x] `RecurrenceResolver.generateOccurrences()` with tests for all supported frequencies
- [x] `fixedExpensePropagationProvider` removed and "expenses copied" SnackBar eliminated
- [x] `ObligationEngine` reads pending `recurring_occurrences` and returns correct obligations
- [x] Automatic detection with precision >= 80% in tests with real data
