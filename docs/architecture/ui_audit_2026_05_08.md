## 🐛 Bug Fixed This Session

### `IntelligenceLayer.analyze()` — `..take(maxVisible)` wasn't truncating list

**File**: `lib/core/domain/services/intelligence_layer.dart`

Code used cascade `..take(maxVisible)` which calls `take()` but discards return — list was never truncated to 3 insights.

```dart
// ❌ ANTES (bugado) — cascade descarta o resultado de take()
return raw
    .where(...)
    .toList()
  ..sort(_byPriority)
  ..take(maxVisible);  // ← retorno ignorado!

// ✅ DEPOIS (corrigido)
final filtered = raw
    .where(...)
    .toList()
  ..sort(_byPriority);
return filtered.take(maxVisible).toList();
```

**Impacto**: O `InsightsPanel` recebia potencialmente N insights (sem limite), saturando a UI em casos com muitas regras disparando simultaneamente. A limitação a 3 agora funciona corretamente.

## 🟡 O que precisa de migração

### 1. `health_screen.dart` — Recalcula o healthScore manualmente

**Severidade**: Média — funciona, mas duplica lógica que já existe no `FinancialEngine`

**Problema**: A tela busca dados brutos separados e chama `FinancialCalculatorService.calculateHealthScore()` diretamente, em vez de usar `financialSnapshotProvider.healthScore` que já o computou.

```dart
// ❌ ATUAL — 4 providers + cálculo manual
final cash = ref.watch(cashExpensesProvider);          // double
final byCategory = ref.watch(cashExpensesByCategoryProvider);
final balance = ref.watch(cashRemainingProvider);
final inst = ref.watch(installmentsProvider).value ?? []; // CardInstallment legado!
final score = FinancialCalculatorService.calculateHealthScore(...);

// ✅ DEVERIA SER
final snap = ref.watch(financialSnapshotProvider);
final score = snap.healthScore; // já calculado, consistente com o dashboard
```

**Nota extra**: `installmentsProvider` usa `CardInstallment` (modelo legado). O `healthScore` já incorpora os dados corretos via `FinancialEngine` que recebe `InstallmentPlan`.

**Ação**: Substituir os 4 providers por `financialSnapshotProvider` e remover o cálculo manual.

---

### 2. `expense_breakdown.dart` (widget do dashboard) — `categoriesMapProvider` retorna `Category`, não `CategoryRef`

**Severidade**: Baixa — funciona, mas usa o modelo `Category` em vez do value object `CategoryRef`

**Problema**: O widget usa `categoriesMapProvider` → `Map<String, Category>` para exibir nomes/emojis de categorias. O sistema novo usa `CategoryRef`. Inconsistência pode causar gaps quando categorias custom não aparecem no mapa.

```dart
// ❌ ATUAL
final catsMap = ref.watch(categoriesMapProvider); // Map<String, Category>

// ✅ DEVERIA SER
final cats = ref.watch(categoriesRefProvider); // List<CategoryRef>
final catsMap = { for (final c in cats) c.slug: c };
```

**Ação**: Trocar `categoriesMapProvider` por `categoriesRefProvider` no widget e nos builds que usam `catsMap`.

---

### 3. `quick_add_bottom_sheet.dart` e `edit_expense_bottom_sheet.dart` — Category picker usa `categoriesStreamProvider`

**Severidade**: Média — se uma categoria custom não existir no stream local, não aparece no picker

**Problema**: Os dois bottom sheets que permitem criar/editar gastos usam `categoriesStreamProvider` → `List<Category>` (Supabase stream direto) em vez de `categoriesRefProvider` que é o source-of-truth do domínio e tem o fallback `CategoryRef.uncategorized()`.

```dart
// ❌ ATUAL (ambos os arquivos)
final categoriesAsync = ref.watch(categoriesStreamProvider);
final categories = ref.read(categoriesStreamProvider).value ?? [];

// ✅ DEVERIA SER
final categories = ref.watch(categoriesRefProvider);
```

**Ação**: Trocar `categoriesStreamProvider` por `categoriesRefProvider` nos dois arquivos. O `CategoryRef` já tem `id`, `slug`, `name`, `emoji` — tudo que o picker precisa.

## 🔴 Padrão Legado ainda exposto

### 4. `transactions_screen.dart` — `isFixed` / `deleteFixedSeriesFrom` ainda na UI

**Severidade**: Média — o padrão antigo (`isFixed + copy`) ainda é utilizável pelo usuário

**Problema**: Na tela de transações, ao fazer swipe de delete em um gasto marcado como `isFixed`, o app ainda oferece a opção "Excluir toda a série" que chama `deleteFixedSeriesFrom()`. Este método busca expenses por `is_fixed=true` e `category` — o padrão antigo que deveria ter sido substituído por `RecurringRules`.

```dart
// transactions_screen.dart linha ~967
if (isFixed) {
  await ref.read(expenseRepositoryProvider).deleteFixedSeriesFrom(expense);
}
```

O método `deleteFixedSeriesFrom` no repositório:
```dart
// expense_repository.dart linha ~184
Future<void> deleteFixedSeriesFrom(Expense from) async {
  // Busca expenses por is_fixed=true, category, payment_method
  // Deleta todos — ignora RecurringRules completamente
}
```

**Ação**: Verificar se ainda existem expenses com `is_fixed=true` em produção (pós-migração). Se não, remover a branch de UI e o método. Se sim, mostrar aviso ao usuário para converter em RecurringRule antes de excluir.
