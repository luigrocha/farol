## 🔍 Problem Context

### Current state (confirmed in code)

```dart
// ONLY recurrence mechanism that existed:
// DashboardScreen.dart — on opening a new month, copies expenses with isFixed = true

ref.listen<AsyncValue<int>>(fixedExpensePropagationProvider, (_, next) {
  next.whenData((count) {
    if (count > 0) {
      ScaffoldMessenger.of(context).showSnackBar(
        SnackBar(content: Text('$count fixed expense(s) copied from previous month')),
      );
    }
  });
});

// Expense with isFixed = true
// This was the "recurrence engine" — one row copy per month
```

### Concrete problems with `isFixed + copy`

1. **Variable amount**: Rent went up R$100 in March — but the copy uses the old amount
2. **No alternative frequencies**: Netflix is monthly but car insurance is semi-annual — impossible in the current model
3. **No pause**: To pause a recurring, you have to delete and recreate
4. **No start/end date**: A recurring that starts in 3 months cannot be planned
5. **No automatic detection**: The system doesn't identify that you pay "Netflix R$45" every month
6. **No real forecasting**: Future recurrings do not impact the balance projection
7. **SnackBar as UX**: Notifying copy with SnackBar is emergency UX, not product UX

### Why this blocks Forecasting

The `ObligationEngine` needs to know: "in the next 3 months, what are the confirmed recurring expenses?". With `isFixed + copy`:
- Future recurrings don't exist until the month arrives
- The balance projection ignores R$2,000+/month in fixed expenses (rent, subscriptions, etc.)

## 📐 Recurring Model

### Concept: RecurringRule + RecurringOccurrence

```
RecurringRule (the rule — the "template" of the recurring)
  │ name: 'Rent', category: housing, amount: R$2,500, frequency: monthly, day: 10
  │ starts_on: 2025-01-10, ends_on: null (indefinite)
  │ status: active
  │
  └─── RecurringOccurrence (each instance generated by the rule)
         scheduled_date: 2026-02-10, amount: R$2,500, status: pending
         scheduled_date: 2026-03-10, amount: R$2,500, status: pending
         scheduled_date: 2026-04-10, amount: R$2,500, status: pending
                │
                └── Expense (created when marked as paid)
                      transaction_date: 2026-02-10
                      amount: R$2,500
                      recurring_rule_id: <uuid>
                      recurring_occurrence_id: <uuid>
```

### Supabase Schema

```sql
-- Recurrence rules
CREATE TABLE recurring_rules (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID REFERENCES auth.users(id) NOT NULL,
  category_id     UUID REFERENCES categories(id),

  -- Identification
  name            TEXT NOT NULL,
  description     TEXT,

  -- Amount
  base_amount     NUMERIC(12,2) NOT NULL CHECK (base_amount > 0),
  amount_type     TEXT NOT NULL DEFAULT 'fixed'
    CHECK (amount_type IN ('fixed', 'variable', 'range')),
  amount_min      NUMERIC(12,2),  -- for range
  amount_max      NUMERIC(12,2),  -- for range

  -- Frequency rule
  frequency       TEXT NOT NULL
    CHECK (frequency IN ('weekly', 'biweekly', 'monthly', 'quarterly', 'semiannual', 'yearly')),
  interval_count  INT NOT NULL DEFAULT 1,  -- every N frequencies
  day_of_month    INT CHECK (day_of_month BETWEEN 1 AND 28),  -- for monthly
  month_of_year   INT[],                   -- for semiannual/yearly (e.g. [1, 7])

  -- Rule validity period
  starts_on       DATE NOT NULL,
  ends_on         DATE,                    -- null = indefinite
  ends_after_n    INT,                     -- or ends after N occurrences

  -- State
  status          TEXT NOT NULL DEFAULT 'active'
    CHECK (status IN ('active', 'paused', 'cancelled')),
  paused_until    DATE,                    -- automatic reactivation

  -- Payment method
  payment_method  TEXT,

  -- Automatic detection
  is_auto_detected      BOOLEAN DEFAULT FALSE,
  detection_confidence  NUMERIC(4,3),     -- 0.0 – 1.0

  -- Legacy isFixed migration
  legacy_expense_id INT,                  -- original expense with isFixed=true

  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Occurrences generated by the rule
CREATE TABLE recurring_occurrences (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  rule_id         UUID REFERENCES recurring_rules(id) ON DELETE CASCADE NOT NULL,
  user_id         UUID REFERENCES auth.users(id) NOT NULL,

  -- When and how much
  scheduled_date  DATE NOT NULL,
  expected_amount NUMERIC(12,2) NOT NULL,

  -- State
  status          TEXT NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending', 'paid', 'skipped', 'overridden')),
  paid_date       DATE,
  actual_amount   NUMERIC(12,2),

  -- Link to the real transaction
  expense_id      UUID REFERENCES expenses(id),

  -- Exception: this occurrence was manually modified
  is_exception    BOOLEAN DEFAULT FALSE,
  exception_notes TEXT,

  created_at      TIMESTAMPTZ DEFAULT NOW(),

  UNIQUE(rule_id, scheduled_date)
);

-- Index for ObligationEngine
CREATE INDEX idx_recurring_occurrences_pending
  ON recurring_occurrences(user_id, scheduled_date, status)
  WHERE status = 'pending';
```

## 🗺️ Incremental Strategy

### PHASE 1 — RecurrenceResolver (pure Dart, no UI) ✅ COMPLETE
**Goal**: The heart of the logic — generate occurrences from a rule.
**Reversibility**: 100% — new code, not touching existing.

```
Task 1.1: RecurringRule entity
  - All schema fields
  - nextOccurrenceDate() computed
  - isActiveOn(date) computed
  - Unit tests for nextOccurrenceDate() for all frequencies

Task 1.2: RecurrenceResolver.generateOccurrences(rule, dateRange)
  - Algorithm: starts at firstOccurrenceInOrAfter(range.start)
  - Loop until range.end or ends_on or ends_after_n
  - Returns List<RecurringOccurrence> with status = pending
  - Edge cases:
    → dayOfMonth > days in month (e.g. day 31 in February) → last day of month
    → ends_after_n: counts occurrences, not days
    → paused_until: skips occurrences until reactivation date

Task 1.3: RecurrenceResolver tests
  - Monthly day 10 for 12 months → 12 correct occurrences
  - Semiannual (jan + jul) → 2 occurrences per year
  - Ends after 6 occurrences → exactly 6
  - Paused until March → jan/feb occurrences absent
```

**Success test**: `dart test test/core/recurrence_resolver_test.dart` → all pass.

### PHASE 3 — Migration of existing isFixed expenses ✅ COMPLETE
**Goal**: Convert `Expense.isFixed = true` to `RecurringRule`.

```
Task 3.1: Detection of existing fixed expenses
  - Query: SELECT DISTINCT category, store_description, amount, payment_method
    FROM expenses WHERE is_fixed = true
  - Group by (category, store_description) to identify unique recurrings

Task 3.2: Create RecurringRules for each group
  - frequency: monthly (all existing are monthly)
  - day_of_month: mode of historical payment day
  - base_amount: median of historical amounts
  - legacy_expense_id: id of the most recent expense in the group

Task 3.3: Remove fixedExpensePropagationProvider
  - Replace automatic copy with:
    → RecurringEngine generates occurrences for the next 3 months on init
    → Dashboard shows "X recurring expenses scheduled" instead of copying expenses
  - Remove SnackBar "expenses copied"

Task 3.4: Mark historical expenses as paid occurrences
  - For each expense with is_fixed = true in history:
    → Create RecurringOccurrence with status = paid, expense_id = expense.id
    → This provides real history for forecasting from the beginning
```

### PHASE 5 — Automatic Detection ✅ COMPLETE
**Goal**: The engine detects recurring patterns in history.

```
Task 5.1: RecurringDetector.detect(List<Expense> history)
  - Group by (store_description, category, ~amount)
    → ~amount: ±5% variation is considered the same recurring
  - For each group with >= 3 occurrences:
    → Calculate intervals between dates
    → Detect dominant frequency (monthly, weekly, etc.)
    → Calculate consistency (standard deviation of intervals)
    → confidence = consistency × (1 - value_variance × 0.5)
  - Return List<RecurringRuleCandidate> with confidence >= 0.75

Task 5.2: Suggestion UI (non-invasive)
  - Discrete card in dashboard: "Found 3 possible recurrings"
  - Tap → suggestions screen with [Confirm] [Edit] [Ignore]
  - On confirm: creates RecurringRule + marks historical occurrences as paid
  - On ignore: don't ask again for 30 days for that pattern

Task 5.3: Detector limits
  - Minimum 3 occurrences with confidence >= 0.75 to suggest
  - Don't suggest if user already has a similar RecurringRule
  - Don't interrupt the expense entry flow
```

## 🚨 Risks and Mitigations

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Migration of isFixed generates duplicate RecurringRules | High | Dirty data | Group by (category, store_description) before creating |
| RecurrenceResolver with calendar bugs (e.g. February) | High | Wrong dates | Exhaustive edge case tests |
| Detector suggests false-positive recurrings | Medium | Distrustful user | High confidence threshold (0.75) + "always ignore" option |
| Too many occurrences generated (endless rule × 3 months) | Medium | Excessive storage use | Generate only 3 months ahead; regenerate on demand |
| `fixedExpensePropagationProvider` removed before RecurringEngine is stable | High | Fixed expenses disappear | Remove only in Phase 3, after RecurringEngine is validated |
